The 2026-06-09 worst-case follow-up also added Path 3 (two-phase-to-optimal) in `src/rubik_optimal/solvers/kociemba_optimal.py` (`solve_kociemba_two_phase_optimal`). It iterates Kociemba phase-1 maneuver depth, pairs each distinct reachable G1 subgroup state with an OPTIMAL phase-2 solve, and terminates when the phase-1 depth reaches the best total found. It is provably optimal by a subgroup-decomposition argument (because G1 is closed under its generators, every solution splits into a phase-1 maneuver plus a phase-2 maneuver of equal total length, so enumerating phase-1 maneuvers + optimal phase-2 is complete; the proof is documented in the module). The implementation is conservative about the optimality claim: it distinguishes a proven phase-2 `lower_bound` (no solution within the cutoff) from an inconclusive `timeout`/`node_limit`, and it returns `status="exact"` ONLY when optimality was actually proven within the budget — otherwise it returns the verified solution as `non_exact` with the proven lower bound. Its honest limitation: a pure-Python iterated two-phase search is tractable only for moderate depths (it proves optimality for depth <=~12 in seconds to ~30s; on a depth-14 state within a 60 s budget it correctly returns a verified non-optimal solution rather than claiming `exact`). It is therefore an INDEPENDENT optimal cross-check for the native Korf solver and the thesis-legitimate "to-optimal mode" of the required Kociemba algorithm, but it does NOT extend the practical optimal-solving depth frontier and does NOT crack the depth-20 superflip quickly.
